# ADR-001 — Architecture Decision Record
**Title:** Client-only Next.js + localStorage for FocusDo MVP  
**Status:** ACCEPTED  
**Date:** 2025-07-07  
**Deciders:** Founding team  

---

## Context

We are building a keyboard-first todo app MVP whose primary goal is to validate three
hypotheses (H1, H2, H3) around speed, keyboard usage, and stability. We need to choose:

1. **Frontend framework**
2. **Data persistence strategy**
3. **Analytics approach**
4. **Deployment target**

The validation window is 48 hours. Speed-to-production and zero backend ops overhead
are the highest-priority constraints.

---

## Decision Drivers

| Driver | Weight |
|--------|--------|
| Ship in < 1 day | High |
| No backend ops burden | High |
| Hypothesis metrics must be measurable | High |
| Cost at zero users | Low (must be free) |
| Scalability beyond 10k users | **Not a concern for MVP** |

---

## Options Considered

### Option A — Next.js (App Router) + localStorage ✅ CHOSEN
- Pure client-side data; no API routes needed
- Vercel free tier; deploys in ~2 min
- localStorage is the analytics ring-buffer as well as task store
- Limitation: data is device-local; acceptable for 48 h validation

### Option B — Next.js + Supabase
- Adds ~4 h of infra setup, auth, RLS policies
- Overkill; no multi-device requirement in H1-H3
- Rejected: timeline risk

### Option C — Vite + React (no Next.js)
- Simpler, but Vercel deploys Next.js first-class; less friction
- No SSR/SEO advantage needed but no disadvantage either
- Rejected: marginal benefit, less ecosystem support

### Option D — Remix + SQLite (fly.io)
- Real database from day 1
- Significant ops overhead, cold-start latency
- Rejected: violates "no backend ops" driver

---

## Decision

**Chosen: Option A**

Next.js 14 App Router renders a fully client-interactive SPA page.
`localStorage` handles both task persistence and analytics event buffering.
No API routes, no database, no auth layer.

The analytics module (`lib/analytics.ts`) stores events in a 500-event ring buffer
(`focusdo:events` key). `getStats()` computes H1/H2/H3 metrics on read.

---

## Consequences

**Positive:**
- Zero backend cost; deploy to Vercel free tier
- Full type safety via TypeScript throughout
- Metrics visible in-browser without any external service
- Can be replaced with a real backend in Phase 2 by swapping `lib/analytics.ts`'s
  `flushToBackend()` stub and `lib/tasks.ts`'s `saveTasks` for API calls

**Negative:**
- Data does not survive a browser clear or a different device
- Event ring-buffer is lossy (>500 events discards oldest)
- No real-time collaboration possible

**Mitigation for Phase 2:**
- Add `POST /api/events` endpoint; keep client-side ring-buffer as write-ahead log
- Add Postgres + Drizzle ORM for task persistence
- Add Clerk/NextAuth for authentication

## Component Map

```
app/
  page.tsx             ← root SPA page, composes everything
  layout.tsx           ← metadata + global CSS
  globals.css          ← CSS variables, base reset

components/
  TaskInput.tsx        ← new-task capture UI
  TaskItem.tsx         ← single task row (edit, complete, delete)
  FocusMode.tsx        ← 3-slot focus view
  HelpModal.tsx        ← keyboard shortcut reference
  Dashboard.tsx        ← H1/H2/H3 live metrics

hooks/
  useTodos.ts          ← task state machine + localStorage persistence
  useKeyboard.ts       ← global keydown handler + shortcut dispatch

lib/
  types.ts             ← Task, AppEvent, HOTKEY_MAP (canonical types)
  tasks.ts             ← pure task mutators + selectors
  analytics.ts         ← event tracking, ring-buffer, getStats()
```
