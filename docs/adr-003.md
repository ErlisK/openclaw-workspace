# ADR-003 — Production Infrastructure: CI/CD, Observability & Reliability
**Title:** Production-readiness stack for FocusDo 48-hour hypothesis validation  
**Status:** ACCEPTED  
**Date:** 2025-07-07  
**Follows:** ADR-002 (Supabase + PostHog + Sentry + PWA)

---

## Context

Before opening the 48-hour hypothesis validation window (H1/H2/H3), we need:
1. Automated quality gates (no broken code reaches production)
2. Observable error rate (H3 requires < 1% over 48h)
3. Performance guarantees (P90 < 1.5s)
4. Uptime monitoring (know immediately if it breaks)
5. Data safety (backup strategy for Supabase Postgres)

---

## Decisions

### CI/CD: GitHub Actions ✅

**Why GitHub Actions over alternatives (Vercel CI, CircleCI, Jenkins):**
- Free for public repos; unlimited minutes on runner
- Co-located with source code (no separate service)
- Vercel CLI works perfectly in Ubuntu runners
- Concurrency controls prevent duplicate deploys

**Pipeline:**
```
PR opened          → CI (typecheck + lint + unit + build + smoke) + Preview Deploy
PR merged to main  → CI + Production Deploy + Lighthouse
Every 15 min       → Uptime check
Daily 03:00 UTC    → Backup verification + H3 audit
```

### Unit Testing: Vitest ✅

**Why Vitest over Jest:**
- Native ESM support (no transform config needed for TypeScript)
- 10× faster than Jest in watch mode
- Same API as Jest (zero migration cost)
- Built-in coverage via V8 (no istanbul config)

**Coverage thresholds:** Lines 60%, Functions 60%, Branches 50%  
(intentionally low for MVP — raise to 80% in Phase 2)

### E2E Testing: Playwright ✅

**Why Playwright over Cypress:**
- Headless by default (faster in CI)
- Better network intercept API for future Supabase mocking
- Free OSS; first-class GitHub Actions support

**Scope:** Smoke tests only (critical path) — full regression suite in Phase 2

### Performance Budget: Lighthouse CI ✅

**Budget targets (`.lighthouserc.json`):**

| Metric | Budget | Rationale |
|--------|--------|-----------|
| LCP | ≤ 2 500 ms | Core Web Vital; aligns with P90 < 1.5s product target |
| FCP | ≤ 1 800 ms | First perceived response |
| TTI | ≤ 3 500 ms | Interactive for keyboard users |
| TBT | ≤ 300 ms | Input responsiveness (critical for keyboard-first UX) |
| CLS | ≤ 0.1 | Layout stability |
| Performance score | ≥ 85 | Overall |

**Note:** P90 < 1.5s refers to task completion median (H1 metric), not page load.  
Page load P90 target is < 2.5s LCP.

### Security Headers: vercel.json ✅

Configured: CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.

**CSP allowlist rationale:**
- `unsafe-inline` for Next.js inline styles (required; remove in Phase 2 with nonce)
- `unsafe-eval` for Next.js dev mode (remove in production Phase 2)
- Supabase, PostHog, Sentry domains explicitly allowlisted

### Uptime Monitoring: GitHub Actions Cron ✅ (MVP) → BetterStack (Phase 2)

**Why GitHub Actions for MVP:**
- Free; already in use; 15-minute granularity is sufficient for 48h test
- Auto-creates GitHub issues on failure (visible to team)

**Phase 2:** Move to BetterStack/UptimeRobot for:
- Sub-5-minute polling
- SMS/call alerts (not just issue creation)
- Status page for users

### Backup Strategy: Supabase Built-in + Manual pg_dump ✅

**Supabase Free tier:**
- No automatic PITR
- Manual approach: `pg_dump $DATABASE_URL | gzip > backup-$(date).sql.gz` weekly

**Supabase Pro tier (Phase 2):**
- Automatic daily backups, 7-day retention
- Point-in-time recovery (PITR)

**Verification:** `maintenance.yml` workflow checks Supabase project is reachable daily.

## Alternatives Rejected

| Alternative | Rejected reason |
|-------------|----------------|
| Vercel's built-in CI | Can't run unit tests or coverage |
| CircleCI | Another service to manage; GitHub Actions is sufficient |
| DataDog APM | Expensive; PostHog + Sentry covers MVP needs |
| PagerDuty | Overkill for single-team MVP |
| Cypress | Slower in CI; Playwright is superior |
| Jest | Slower than Vitest; ESM config overhead |

## Consequences

**Positive:**
- Zero broken deploys reach production (CI gates)
- Hypothesis window is observable (error rate, performance, uptime)
- Security headers shipped from day 1 (not retrofitted)
- All checks are automated and free for MVP scale

**Negative:**
- GitHub Actions cron minimum is 5 minutes (not 1-minute uptime SLA)
- No SMS/call alerts on downtime (Phase 2)
- Lighthouse CI requires a separate LHCI token for PR status checks
- `unsafe-eval` in CSP required for Next.js dev (acceptable for MVP)
