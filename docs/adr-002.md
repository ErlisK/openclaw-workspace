# ADR-002 — Full Stack Architecture: Supabase + PostHog + Sentry + PWA
**Title:** Production stack upgrade for authenticated, observable FocusDo  
**Status:** ACCEPTED  
**Date:** 2025-07-07  
**Supersedes:** ADR-001 (localStorage-only)  
**Deciders:** Founding team

---

## Context

Phase 1 validated the UI and hypothesis measurement loop using localStorage.
Phase 2 (validation) requires:

1. **User identity** — to attribute H1/H2/H3 metrics per-user across sessions/devices
2. **Real analytics** — PostHog for funnel analysis, retention, shortcut heatmaps
3. **Error observability** — Sentry for H3 (< 1% error rate) validation with real stack traces
4. **Cross-device persistence** — Supabase Postgres replaces localStorage for task storage
5. **Offline resilience** — PWA service worker + localStorage as write-ahead log

---

## Decision Drivers

| Driver | Weight |
|--------|--------|
| Time to ship (< 4 h integration) | High |
| Per-user analytics (H1/H2 attribution) | High |
| Real error rate tracking (H3) | High |
| Free tier for < 1 000 MAU | High |
| Type safety throughout | Medium |
| No vendor lock-in for core data | Medium |

## Stack Decision

### Auth + Database: Supabase ✅

**Why Supabase over alternatives:**

| Alternative | Rejected Reason |
|-------------|----------------|
| Firebase | NoSQL; no SQL aggregations for H1/H2/H3 views; vendor lock-in |
| PlanetScale | MySQL only; no RLS; requires custom auth |
| Auth0 + PlanetScale | Two vendors, more credentials, slower iteration |
| Clerk + Drizzle + Neon | Best long-term but 3× setup time vs Supabase |
| **Supabase** | ✅ Auth + Postgres + RLS + client SDK in one; generous free tier |

**Key Supabase features used:**
- `auth.users` — magic link + email/password sign-in (zero custom auth code)
- `public.tasks` with RLS — users only ever see their own data
- `public.events` — analytics backup/audit trail
- Realtime subscriptions — Phase 3 (deferred)

### Analytics: PostHog ✅

**Why PostHog over alternatives:**

| Alternative | Rejected Reason |
|-------------|----------------|
| Amplitude | No self-host option; expensive at scale |
| Mixpanel | Same; 100k event free limit too restrictive |
| Supabase built-in analytics | Basic; no funnel/cohort; no session replay |
| GA4 | No custom events without complexity; privacy concerns |
| **PostHog** | ✅ GDPR-friendly; open source; funnels + heatmaps; generous free tier; cloud-hosted |

**Events tracked:**
- `session_started` — H3 denominator; activation funnel step 1
- `task_created` — H1 start clock; funnel step 2; H2 creation method
- `task_completed` — H1 `time_to_complete_ms`; H2 `input_method`; funnel step 3
- `focus_mode_toggled` — feature adoption; context for H1 segmentation
- `keyboard_shortcut_used` — H2 surrogate; shortcut adoption heatmap
- `error_caught` — H3 error rate numerator

**PostHog config decisions:**
- `autocapture: false` — manual events only (keyboard-first UX; avoid noise)
- `disable_session_recording: true` — Phase 2 (requires consent UX)
- `capture_pageview: false` — we fire `session_started` manually

### Error Tracking: Sentry ✅

**Why Sentry:**
- De-facto standard; Next.js first-class integration (`@sentry/nextjs`)
- Source maps upload automatically on Vercel deploy
- H3 metric (`< 1% error rate`) needs real stack traces, not just counts

**Config:**
- `tracesSampleRate: 0.1` in production (performance budget)
- Ignore `ResizeObserver` and `Non-Error promise rejection` noise
- Tags: `phase=mvp-1`, `hypothesis_window=48h`

### PWA Baseline ✅

**Why now:**
- Offline task capture is table-stakes for productivity apps
- Service worker = localStorage write-ahead log survives poor connectivity
- Manifest enables "Add to Home Screen" without App Store friction

**Scope (Phase 2 MVP):**
- Cache-first for static assets
- Network-first with offline fallback for navigation
- `manifest.json` with theme color + icons
- **Deferred:** Background sync, push notifications, Workbox

## Data Flow

### Task creation
```
User presses N → types text → Enter
  ↓
useTodos.addTask()
  ├── lib/tasks.ts: createTask() [pure, in-memory]
  ├── lib/supabase.ts: tasks.insert() [if authenticated]
  ├── lib/tasks.ts: saveTasks() [localStorage always]
  └── lib/analytics.ts: track("task_created")
        ├── localStorage ring-buffer
        └── PostHog.capture("task_created", { input_method, text_length })
```

### Analytics read (Dashboard)
```
User opens 📊
  ↓
Dashboard component
  ├── getStats() ← localStorage (always fast, no network)
  └── Phase 2: supabase.from("v_hypothesis_summary").select()
```

## Graceful Degradation

All three integrations are **optional at runtime**:
- No Supabase URL → app works in localStorage-only mode (Phase 1 behaviour)
- No PostHog key → events stored in localStorage ring-buffer only
- No Sentry DSN → errors not forwarded externally (console only in dev)

This ensures the MVP stays deployable without any credentials.

---

## Consequences

**Positive:**
- Full observability stack for 48 h hypothesis validation
- Zero server infra (Supabase manages Postgres; Vercel manages compute)
- Offline-capable with PWA + localStorage fallback
- Incremental: each integration is independently optional

**Negative:**
- Three new external services to configure (Supabase, PostHog, Sentry)
- Fine-grained PAT token limitations may block GitHub push (infra issue, not architecture)
- PostHog free tier: 1M events/month (not a concern at MVP scale)

**Phase 3 additions (not in scope now):**
- Supabase Realtime for cross-device task sync
- PostHog session replay (requires consent UI)
- Supabase Edge Functions for server-side event ingestion
- Workbox for advanced PWA caching
